# 3가지 이상 대안 요청하기

## 핵심 요약

판단하기 어려운 상황에서는 Claude Code에게 3가지 이상의 대안을 제시해달라고 하세요. 각 대안의 장단점을 비교하고 최선의 선택을 할 수 있습니다.

## 왜 대안을 요청해야 하나?

### 대안 없이 진행 vs 대안 비교

```
❌ 대안 없이 진행:
"로그인 기능 만들어줘"
→ Claude Code가 하나의 방법으로 구현
→ 더 좋은 방법이 있었는지 모름
→ 나중에 변경 필요할 수도

✅ 대안 비교 후 진행:
"로그인 기능 구현 방법 3가지 제안해줘"
→ 각 방법의 장단점 파악
→ 프로젝트에 맞는 최선의 선택
→ 의사결정 근거 확보
```

### 대안 요청이 유용한 상황

| 상황 | 예시 |
|------|------|
| 기술 선택 | 어떤 라이브러리/프레임워크를 쓸지 |
| 아키텍처 결정 | 어떤 패턴/구조를 적용할지 |
| 문제 해결 | 버그를 어떤 방식으로 고칠지 |
| UI/UX 디자인 | 어떤 레이아웃/인터랙션을 적용할지 |
| 성능 최적화 | 어떤 방법으로 개선할지 |

## 대안 요청 방법

### 기본 요청

```
"로그인 기능 구현 방법 3가지 제안해줘. 각각 장단점 설명해줘"
```

### 상세 요청

```
"회원 데이터 저장 방법을 제안해줘:
- 최소 3가지 대안
- 각각의 장점과 단점
- 구현 복잡도
- 확장성
- 보안 측면

우리 프로젝트에 가장 적합한 것 추천해줘"
```

### 특정 기준 요청

```
"인증 방식 대안 3가지 알려줘:
- 보안성 관점
- 사용자 편의성 관점
- 구현 난이도 관점에서 비교해줘"
```

## 대안 비교표 요청

### 표 형식으로 요청

```
"상태 관리 라이브러리 옵션을 표로 비교해줘:
- Redux, Zustand, Context API, Recoil
- 학습 곡선, 번들 크기, 기능, 커뮤니티 규모 비교"
```

### 응답 예시

| 라이브러리 | 학습 곡선 | 번들 크기 | 기능 | 커뮤니티 |
|-----------|----------|----------|------|----------|
| Redux | 높음 | 큼 | 풍부 | 매우 큼 |
| Zustand | 낮음 | 작음 | 적당 | 성장 중 |
| Context API | 낮음 | 없음 | 기본 | - |
| Recoil | 중간 | 중간 | 풍부 | 성장 중 |

## 활용 시나리오

### 시나리오 1: 데이터베이스 선택

```
"프로젝트 데이터베이스 선택지를 3가지 이상 제안해줘:

프로젝트 특성:
- 사용자 수: 초기 1,000명, 성장 예상
- 데이터: 주로 JSON 형태
- 쿼리: 복잡한 조회보다 단순 CRUD 위주

각 옵션의 장단점과 비용도 알려줘"
```

### 시나리오 2: 버그 수정 방법

```
"이 버그를 고치는 방법 3가지 알려줘:
1. 빠른 임시 해결책
2. 정석적인 해결책
3. 근본 원인 해결책

각각 얼마나 걸리고 어떤 장단점이 있는지"
```

### 시나리오 3: UI 레이아웃

```
"대시보드 레이아웃 대안 3가지 제안해줘:
- 정보 밀도 높은 버전
- 깔끔하고 간결한 버전
- 모바일 친화적인 버전

각각 어떤 사용자에게 적합한지 설명해줘"
```

## 결정 후 진행

### 선택 후 지시

```
"2번 방식(JWT + Refresh Token)으로 진행해줘"

"A안과 B안을 조합해서 구현해줘:
- A안의 인증 방식
- B안의 토큰 저장 방식"
```

### 선택 이유 기록 요청

```
"3번 방식으로 결정했어. 왜 이 방식을 선택했는지
Documentation에 기록해줘"
```

## 대안 요청 팁

### 1. 구체적인 맥락 제공

```
❌ "로그인 방법 알려줘"
✅ "소규모 SaaS 프로젝트에서 로그인 방법 3가지 제안해줘.
   사용자 수 1,000명 이하, 보안 중요도 높음"
```

### 2. 비교 기준 명시

```
❌ "어떤 게 좋아?"
✅ "성능, 유지보수성, 학습 비용 측면에서 비교해줘"
```

### 3. 제약 조건 포함

```
❌ "프론트엔드 프레임워크 추천해줘"
✅ "프론트엔드 프레임워크 추천해줘.
   - 팀원 모두 JavaScript 경험만 있음
   - 6개월 내 출시 목표
   - SEO 중요"
```

## 체크리스트

### 대안 요청 시 포함할 내용

- [ ] 필요한 대안 개수 (최소 3개 권장)
- [ ] 프로젝트/상황 맥락
- [ ] 비교 기준 (성능, 비용, 복잡도 등)
- [ ] 제약 조건 (시간, 기술, 예산 등)
- [ ] 원하는 응답 형식 (표, 목록 등)

## 주의사항

- 너무 많은 대안(5개 이상)은 오히려 결정을 어렵게 함
- 각 대안의 장단점을 꼼꼼히 비교
- 완벽한 선택은 없음 - 트레이드오프 이해하기
- 결정 후에는 선택한 이유 기록
- 나중에 변경이 필요할 수 있음을 염두
