# Task를 작게 나누는 기준

> 이 문서는 SAL Grid에서 Task를 적절한 크기로 나누는 기준과 방법을 설명합니다.

---

## 왜 작게 나눠야 하는가

"인증 시스템 전체 구현" 같은 큰 Task는 완료 시점 예측이 어렵고, 문제 발생 시 원인 찾기도 어렵습니다. "회원가입 API 구현"처럼 작은 Task는 완료 기준이 명확하고 테스트와 추적이 쉽습니다.

---

## 분할의 세 가지 기준

### 1. 단일 책임 원칙
```
✅ 좋은 예: "회원가입 API 구현" - 하나의 기능만 담당
❌ 나쁜 예: "인증 시스템 전체 구현" - 여러 기능 혼합
```

### 2. 독립 테스트 가능성
```
✅ 좋은 예: "회원가입 API 구현" - Postman으로 단독 테스트 가능
❌ 나쁜 예: "결제 완료 페이지" - 결제 API 완료되어야 테스트 가능
```

### 3. 명확한 완료 기준
```
✅ 좋은 예: "회원가입 API가 200을 반환한다"
❌ 나쁜 예: "기능이 잘 동작한다" - "잘"의 기준 모호
```

---

## 분할 예시: 사용자 인증 시스템

**분할 전**
```
"사용자 인증 시스템 구현"
```

**분할 후**
```
[Database] S1D1: users 테이블, S1D2: sessions 테이블
[Security] S1S1: Auth Provider 설정, S2S1: 이메일 인증 API
[Backend] S2BA1: 회원가입 API, S2BA2: 로그인 API
[Frontend] S2F1: 회원가입 페이지, S2F2: 로그인 페이지
```

---

## 너무 작게 나누는 것도 문제

```
❌ 너무 작음:
- "로그인 버튼 스타일 변경" - 별도 Task 불필요
- "변수명 리팩토링" - 다른 Task에 포함
```

**적절한 크기 기준**:
- 수십~수백 줄의 코드
- 하나의 명확한 기능/화면 담당
- 독립적으로 테스트 가능

---

## 1 파일 = 1 Task 원칙

```
✅ 허용: Task 1개 → 파일 여러 개
❌ 금지: 파일 1개 → Task 여러 개 (설계 재검토 필요)
```

여러 Task가 하나의 파일을 수정하면 충돌이 발생합니다.

## 체크리스트

### 분할 기준
- [ ] 각 Task가 단일 책임을 가지는가?
- [ ] 각 Task가 독립적으로 테스트 가능한가?
- [ ] 각 Task의 완료 기준이 명확한가?

### 크기 확인
- [ ] 너무 크지 않은가? (여러 기능 혼합)
- [ ] 너무 작지 않은가? (별도 관리 불필요)

### 의존성
- [ ] 의존성이 명시되어 있는가?
- [ ] 순환 의존성이 없는가?
