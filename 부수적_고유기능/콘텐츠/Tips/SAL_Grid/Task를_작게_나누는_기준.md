# Task를 작게 나누는 기준

## 요약
하나의 Task는 명확한 완료 기준이 있고, 독립적으로 테스트 가능한 크기로 나눈다. 이상적인 Task는 하나의 명확한 결과물을 생성한다.

## 상세

### Task 분할 기준

| 기준 | 좋은 예 | 나쁜 예 |
|------|---------|---------|
| 단일 책임 | "회원가입 API 구현" | "인증 시스템 전체 구현" |
| 독립 테스트 | Task만으로 테스트 가능 | 다른 Task 완료를 기다려야 함 |
| 명확한 완료 | "API가 200 반환" | "기능이 잘 동작함" |

### 분할 예시

```
[큰 기능: 사용자 인증 시스템]
    ↓ 분할
├── S2S1: 회원가입 API
├── S2S2: 로그인 API
├── S2S3: 로그아웃 API
├── S2F1: 회원가입 UI
├── S2F2: 로그인 UI
└── S2D1: users 테이블 RLS
```

### 올바른 분할 vs 잘못된 분할

```
❌ 잘못된 분할:
- Task 1: 프론트엔드 전체 구현 (너무 큼)
- Task 2: 버튼 색상 변경 (너무 작음)

✅ 올바른 분할:
- S2F1: 로그인 폼 UI 구현
- S2F2: 회원가입 폼 UI 구현
- S2BA1: 로그인 API 구현
```

### 주의
- 의존성 있는 Task는 순서 명시 필수
- Task 완료 기준은 객관적/측정 가능해야 함
- 너무 세분화하면 관리 오버헤드 증가
