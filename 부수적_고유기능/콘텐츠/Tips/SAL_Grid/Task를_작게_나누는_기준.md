# Task를 작게 나누는 기준

## 핵심 요약

하나의 Task는 명확한 완료 기준이 있고, 독립적으로 테스트 가능한 크기로 나누세요. 이상적인 Task는 1-4시간 내에 완료할 수 있고, 하나의 명확한 결과물을 생성합니다.

## 왜 Task를 적절히 나눠야 하나?

### Task 크기별 문제점

| 크기 | 문제점 | 결과 |
|------|--------|------|
| 너무 큰 Task | 추적 어려움, 완료 기준 모호 | 진행률 파악 불가 |
| 너무 작은 Task | 관리 오버헤드 증가, 의존성 복잡 | 비효율적 |
| 적절한 Task | 명확한 목표, 측정 가능 | 효율적 개발 |

### 적절한 Task 크기의 장점

```
✅ 진행 상황 명확히 파악 가능
✅ 테스트 범위가 명확함
✅ 검증 기준이 명확함
✅ 문제 발생 시 원인 추적 용이
✅ 작업 분배가 쉬움
```

## Task 분할 기준

### 1. 단일 책임 원칙

```
하나의 Task = 하나의 명확한 책임

✅ 좋은 예:
- "회원가입 API 구현"
- "로그인 폼 UI 구현"
- "users 테이블 생성"

❌ 나쁜 예:
- "인증 시스템 전체 구현" (너무 큼)
- "버튼 색상 변경" (너무 작음)
```

### 2. 독립적 테스트 가능 여부

```
Task 완료 후 독립적으로 테스트할 수 있어야 함

체크리스트:
- [ ] 이 Task만으로 테스트가 가능한가?
- [ ] 다른 Task 완료를 기다릴 필요 없는가?
- [ ] 테스트 결과가 명확한가? (Pass/Fail)
```

### 3. 명확한 완료 기준

```
완료 기준이 객관적으로 측정 가능해야 함

✅ 좋은 완료 기준:
- "API가 200 상태 코드 반환"
- "모든 단위 테스트 통과"
- "폼 제출 시 DB에 데이터 저장"

❌ 나쁜 완료 기준:
- "기능이 잘 동작함" (주관적)
- "대략 구현됨" (모호)
```

### 4. 시간 기준

```
이상적인 Task 소요 시간: 1-4시간

시간대별 적절성:
├── 30분 미만: 너무 작음 → 다른 Task와 합치기
├── 1-4시간: 적절함 ✅
├── 4-8시간: 가능 (복잡한 Task)
└── 8시간 초과: 너무 큼 → 분할 필요
```

## Task 분할 실전 가이드

### 기능 단위로 분할

```
[큰 기능: 사용자 인증 시스템]
    ↓ 분할
├── S2S1: 회원가입 API 구현
├── S2S2: 로그인 API 구현
├── S2S3: 로그아웃 API 구현
├── S2S4: 비밀번호 재설정 API 구현
├── S2F1: 회원가입 UI 구현
├── S2F2: 로그인 UI 구현
└── S2D1: users 테이블 RLS 설정
```

### 레이어별로 분할

```
[기능: 게시글 작성]
    ↓ 레이어별 분할
├── Database: 테이블 스키마
├── Backend API: CRUD API
├── Frontend: 작성 폼 UI
└── Testing: 테스트 케이스
```

### 복잡도별 분할

```
[복잡한 기능: 결제 시스템]
    ↓ 복잡도별 분할
├── 1단계: 기본 결제 플로우
├── 2단계: 결제 검증 로직
├── 3단계: 실패 처리 및 재시도
└── 4단계: 영수증 발급
```

## Area별 Task 분할 예시

### Frontend (F) 분할

```
[페이지: 상품 상세 페이지]
├── S2F1: 상품 정보 표시 컴포넌트
├── S2F2: 상품 이미지 갤러리
```

```
[기능: 주문 관리]
├── S2BA1: 주문 생성 API
├── S2BA2: 주문 조회 API
├── S2BA3: 주문 취소 API
├── S2BA4: 주문 상태 변경 API

```

### Database (D) 분할

```
[스키마: 이커머스]
├── S1D1: users 테이블 생성
├── S1D2: products 테이블 생성
├── S1D3: orders 테이블 생성
├── S1D4: 테이블 간 관계 설정

```

## Task 분할 체크리스트

### 분할 전 확인

- [ ] 전체 기능의 범위가 명확한가?
- [ ] 의존 관계를 파악했는가?
- [ ] 각 Task의 책임이 명확한가?

### 분할 후 확인

- [ ] 각 Task가 독립적으로 테스트 가능한가?
- [ ] 각 Task의 완료 기준이 명확한가?
- [ ] Task 간 의존성이 순환하지 않는가?
- [ ] 각 Task가 1-4시간 내에 완료 가능한가?

## Claude Code에게 Task 분할 요청

### 분할 요청 예시

```
"사용자 인증 기능을 SAL Grid Task로 분할해줘:
- Stage: S2
- 회원가입, 로그인, 로그아웃, 비밀번호 재설정 포함
- Frontend, Backend API, Database 각각 분리
- 각 Task별로 Task ID 부여해줘"
```

### 분할 결과 검토 요청

```
"분할된 Task들 검토해줘:
- 각 Task가 독립적으로 테스트 가능해?
- 의존 관계가 올바르게 설정됐어?
- 너무 크거나 작은 Task 있어?"
```

## 잘못된 분할 vs 올바른 분할

### ❌ 잘못된 분할

```
Task 1: 프론트엔드 전체 구현 (너무 큼)
Task 2: 버튼 색상 변경 (너무 작음)
Task 3: API 및 DB 구현 (혼합됨)
```

### ✅ 올바른 분할

```
S2F1: 로그인 폼 UI 구현
S2F2: 회원가입 폼 UI 구현
S2BA1: 로그인 API 구현
S2BA2: 회원가입 API 구현
S2D1: users 테이블 및 RLS 설정
```

## 주의사항

- Task 크기는 프로젝트 특성에 따라 조정
- 의존성 있는 Task는 순서 명시 필수
- Task 완료 기준은 객관적/측정 가능해야 함
- 분할 후에도 전체 기능이 커버되는지 확인
- Task ID 체계를 일관되게 유지
- 너무 세분화하면 관리 오버헤드 증가
