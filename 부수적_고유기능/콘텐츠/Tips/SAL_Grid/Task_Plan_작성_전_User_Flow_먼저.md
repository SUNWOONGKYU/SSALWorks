# Task Plan 작성 전 User Flow 먼저

> 이 문서는 프로젝트 기획 시 User Flow를 먼저 작성해야 하는 이유와 방법을 설명합니다.

---

## 왜 User Flow가 먼저인가

"개발부터 시작"하면 나중에 "어? 이 기능이 없네?"라는 상황이 반복됩니다. User Flow를 먼저 그리면 빠진 기능을 미리 발견할 수 있습니다.

```
1. User Flow 다이어그램 완성
    ↓
2. 프로토타입 제작
    ↓
3. Task Plan 작성
    ↓
4. 개발 시작
```

---

## User Flow 작성 3단계

**1단계: 사용자 유형 정의**
- 신규 사용자, 기존 사용자, 결제 사용자, 관리자

**2단계: 각 사용자별 여정 그리기**
```
[신규 사용자 여정]
랜딩 페이지 → 회원가입 → 이메일 인증 → 로그인 → 대시보드
```

**3단계: 빠진 화면/기능 확인**
- "비밀번호를 잊어버리면?"
- "결제 실패하면 어떤 화면?"

---

## 실제 사례: 기능 누락

**문제 상황**
"OAuth만 있으면 되겠지" → 이메일 가입 사용자 고려 안 함 → 개발 중간에 S2F3(회원가입 UI), S2BA4(회원가입 API) 추가 → Task Plan, Instruction, Order Sheet 전부 수정

**User Flow로 예방**
```
가입 방식 선택
├─ 이메일 가입 → 이메일 입력 → 인증 → 완료
└─ OAuth 가입 → Google/Kakao → 완료
```

처음부터 "이메일 가입" 경로가 필요하다는 것을 알 수 있습니다.

---

## Task Plan 수정의 연쇄 효과

Task Plan 1줄 수정 → 최소 4개 문서 수정 필요:
1. Task Instruction
2. Seed SQL
3. Verification Instruction
4. Order Sheet

**처음에 제대로 하는 것이 나중에 고치는 것보다 100배 쉽습니다.**

---

## "당연히 있겠지" 가정의 위험

```
❌ "결제 웹훅이 있으니 크레딧 충전은 되겠지"
→ 실제로는 결제 수단 등록 화면 필요

❌ "이메일 발송이 있으니 인증은 되겠지"
→ 실제로는 인증 코드 검증, 재발송, 만료 처리 필요
```

가정하지 말고 User Flow에 모든 단계를 명시합니다.

---

## 체크리스트

### User Flow 작성
- [ ] 모든 사용자 유형을 정의했는가?
- [ ] 각 유형별 전체 여정을 그렸는가?
- [ ] 에러 상황 처리 경로를 포함했는가?
- [ ] "당연히 있겠지" 가정 없이 모든 단계를 명시했는가?

### 검토
- [ ] 빠진 화면이 없는가?
- [ ] 빠진 기능이 없는가?
- [ ] 프로토타입으로 검증했는가?
