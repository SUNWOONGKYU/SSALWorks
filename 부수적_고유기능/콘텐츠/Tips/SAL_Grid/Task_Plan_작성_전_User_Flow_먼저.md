# Task Plan 작성 전 User Flow 먼저

## 요약
Task Plan을 세우기 전에 User Flow를 먼저 작성하고 프로토타입을 만들어야 한다. 그래야 "어? 이 기능이 없네?" 상황을 방지할 수 있다.

## 상세

### 올바른 순서

```
1. User Flow 다이어그램 완성
   (신규 사용자, 기존 사용자, 결제, 관리자 플로우)
    ↓
2. 프로토타입 제작
   (모든 화면 UI 구현, 빠진 기능 발견)
    ↓
3. Task Plan 작성
   (User Flow 각 단계를 Task로 변환)
    ↓
4. 개발 시작
```

### 실제 사례: 회원가입 기능 누락

**잘못된 접근:**
"OAuth 있으니까 회원가입은 필요 없겠지" → 회원가입 UI/API 누락

**User Flow 분석 후:**
```
신규 사용자 → 회원가입 → 이메일 인증 → 로그인
```
→ S2F3(회원가입 UI), S2BA4(회원가입 API) 추가 필요

### "당연히 있겠지" 가정 금지

```
❌ 잘못된 생각:
"결제 웹훅 있으니까 충전은 당연히 되겠지"

✅ 올바른 접근:
"크레딧 충전 전에 결제 수단 등록은 어떻게 하지?"
```

### 주의
- Task Plan 1줄 수정 = 최소 4개 문서 수정 (Instruction, Seed SQL, Verification, Order Sheet)
- 처음에 제대로 하는 게 나중에 고치는 것보다 100배 쉽다
