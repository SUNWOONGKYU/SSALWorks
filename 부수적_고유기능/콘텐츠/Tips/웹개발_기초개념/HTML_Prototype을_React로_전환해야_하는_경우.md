# HTML Prototype을 React로 전환해야 하는 경우

> HTML로 만든 프로토타입이 있을 때, 언제 그대로 두고 언제 React로 전환해야 하는지 설명합니다.

---

## HTML 프로토타입이란

HTML 프로토타입은 웹사이트의 화면을 빠르게 만들어보는 방법입니다. 디자인이 확정되기 전에 "대충 이런 느낌"을 보여주는 것이 목적입니다. 로그인 페이지, 대시보드, 마이페이지 같은 화면을 HTML과 CSS만으로 만들어서 실제 서비스처럼 보이게 합니다.

프로토타입 단계에서는 버튼을 눌러도 실제로 로그인되지 않습니다. 데이터베이스에 저장되지도 않습니다. 그냥 화면만 있는 것입니다. "이 버튼을 누르면 이 화면으로 이동한다"는 흐름을 확인하는 용도입니다. 실제 기능은 나중에 개발 단계에서 추가합니다.

HTML 프로토타입의 장점은 빠르게 만들 수 있다는 것입니다. React나 Vue 같은 프레임워크를 몰라도 HTML과 CSS만 알면 화면을 만들 수 있습니다. 수정도 간단합니다. 버튼 위치를 바꾸고 싶으면 HTML 코드 몇 줄만 고치면 됩니다.

또 다른 장점은 비개발자도 이해할 수 있다는 것입니다. 기획자나 디자이너가 HTML 파일을 브라우저에서 열어보면서 "이 부분 수정해주세요"라고 피드백을 줄 수 있습니다. React 코드를 보여주면 이해하기 어렵지만, HTML은 직관적입니다.

---

## HTML로 충분한 경우

모든 웹사이트가 React를 필요로 하는 것은 아닙니다. 오히려 많은 웹사이트는 HTML만으로도 충분합니다.

정적인 콘텐츠가 대부분인 경우입니다. 회사 소개 페이지, 블로그, 문서 사이트처럼 내용이 자주 바뀌지 않고 사용자 입력이 거의 없는 사이트는 HTML이 더 적합합니다. 로딩도 빠르고 검색 엔진에도 잘 노출됩니다. 구글 검색에서 상위에 노출되려면 오히려 단순한 HTML이 유리할 수 있습니다.

간단한 폼만 있는 경우도 마찬가지입니다. 문의하기 폼, 뉴스레터 구독, 이벤트 신청처럼 단순한 입력만 받는다면 굳이 React를 쓸 필요가 없습니다. HTML form 태그와 약간의 JavaScript로 충분히 처리할 수 있습니다.

페이지 수가 적고 구조가 단순한 경우도 해당됩니다. 5페이지 이하의 작은 사이트라면 React의 장점을 느끼기 어렵습니다. 설정하고 빌드하는 데 드는 시간이 오히려 더 클 수 있습니다.

유지보수 인력이 비개발자인 경우도 고려해야 합니다. 콘텐츠를 수정할 사람이 HTML만 아는 수준이라면, React로 만들어두면 수정이 어려워집니다.

---

## React로 전환해야 하는 경우

반대로 다음과 같은 상황에서는 React 전환을 고려해야 합니다.

첫째, 화면이 자주 바뀌어야 할 때입니다. 사용자가 버튼을 누르면 목록이 추가되고, 삭제 버튼을 누르면 항목이 사라지는 식으로 화면이 실시간으로 변해야 한다면 React가 유리합니다. 장바구니에 상품을 담으면 숫자가 올라가고, 좋아요를 누르면 하트가 채워지는 것처럼요. HTML만으로 이런 기능을 구현하려면 복잡한 JavaScript 코드가 필요하고, 코드가 금방 지저분해집니다.

둘째, 같은 UI가 여러 곳에서 반복될 때입니다. 카드 컴포넌트, 버튼, 모달창, 드롭다운, 알림 배너 같은 것들이 여러 페이지에서 똑같이 쓰인다면 React의 컴포넌트 시스템이 효율적입니다. 한 번 만들어두면 어디서든 재사용할 수 있습니다. 디자인을 수정할 때도 한 곳만 고치면 전체에 반영됩니다.

셋째, 서버와 데이터를 주고받는 기능이 많을 때입니다. 로그인 상태 관리, 장바구니 업데이트, 실시간 알림, 무한 스크롤, 검색 자동완성 같은 기능은 React의 상태 관리가 도움이 됩니다. 데이터가 바뀌면 화면이 자동으로 업데이트되는 React의 특성이 이런 기능 구현을 쉽게 만들어줍니다.

넷째, 팀에서 React 경험이 있을 때입니다. 이미 React를 다룰 줄 아는 개발자가 있다면 React로 개발하는 것이 더 효율적일 수 있습니다. 익숙한 도구로 작업하면 속도가 빠릅니다.

다섯째, 앱처럼 동작해야 할 때입니다. 페이지 전환 시 깜빡임 없이 부드럽게 넘어가고, 뒤로 가기를 눌러도 상태가 유지되는 것처럼 앱 같은 사용자 경험이 필요하다면 React가 적합합니다. 이런 것을 SPA(Single Page Application)라고 부릅니다.

## SSALWorks 방법론의 접근

SSALWorks에서는 P3 단계(프로토타입)에서 HTML로 화면을 먼저 만듭니다. 이 단계에서는 기능 구현보다 화면 설계에 집중합니다. 로그인 페이지가 어떻게 생겼는지, 버튼이 어디에 있는지, 메뉴 구조가 어떤지를 확정하는 것이 목적입니다.

P3 단계가 끝나면 22개의 HTML 파일과 같은 화면 세트가 완성됩니다. 이 파일들을 보면서 "이 흐름이 맞는지", "이 디자인이 적절한지"를 검토합니다. 수정이 필요하면 HTML 파일만 고치면 됩니다. 빠르게 여러 번 수정할 수 있습니다.

그 다음 S2 단계(개발 1차)에서 필요한 페이지만 React로 전환합니다. 모든 페이지를 전환하는 것이 아니라, 복잡한 상호작용이 필요한 페이지만 선별해서 전환합니다. 로그인, 회원가입, 대시보드처럼 데이터 처리가 많은 페이지가 우선입니다. 이용약관, 개인정보처리방침, FAQ 같은 단순한 안내 페이지는 HTML 그대로 유지할 수 있습니다.

이렇게 하면 불필요한 작업을 줄일 수 있습니다. 처음부터 React로 시작하면 화면 설계가 바뀔 때마다 코드를 많이 수정해야 합니다. HTML 프로토타입으로 화면을 확정한 뒤에 전환하면 재작업이 줄어듭니다.

## 전환 판단 기준

다음 질문에 "예"가 3개 이상이면 React 전환을 고려하세요.

사용자 입력에 따라 화면이 실시간으로 변해야 하나요?
같은 UI 요소가 5개 이상의 페이지에서 반복되나요?
로그인 상태를 여러 페이지에서 공유해야 하나요?
서버에서 데이터를 가져와서 목록을 보여줘야 하나요?
페이지 새로고침 없이 화면이 바뀌어야 하나요?

이 질문들에 대부분 "아니오"라면 HTML로 충분할 가능성이 높습니다. 괜히 복잡한 도구를 쓸 필요가 없습니다. 단순한 것이 좋을 때도 있습니다.

## 전환 시 주의사항

React로 전환할 때 몇 가지 주의할 점이 있습니다.

첫째, 한 번에 모든 페이지를 전환하지 마세요. 가장 복잡한 페이지부터 시작해서 점진적으로 전환하는 것이 좋습니다. 문제가 생겼을 때 원인을 파악하기 쉽습니다.

둘째, HTML 프로토타입의 스타일을 그대로 가져오세요. CSS를 처음부터 다시 작성하면 시간이 두 배로 듭니다. 기존 CSS 파일을 그대로 import하거나 스타일 값을 복사하세요.

셋째, 컴포넌트 분리를 과하게 하지 마세요. 처음에는 큰 단위로 시작해서 필요할 때 분리하는 것이 좋습니다. 너무 잘게 나누면 오히려 복잡해집니다.

넷째, 기존 HTML의 동작을 먼저 기록하세요. 전환 전에 어떤 동작이 있었는지 메모해두고, 전환 후에도 같은 동작이 되는지 확인하세요. 전환 과정에서 기능이 빠지는 실수를 방지할 수 있습니다.

---

## 결론

HTML 프로토타입은 빠른 화면 설계에 유용합니다. 디자인 확정 전에 여러 번 수정하기 좋습니다. 비개발자도 피드백을 주기 쉽습니다.

하지만 복잡한 상호작용, UI 재사용, 상태 관리가 필요하다면 React 전환이 필요합니다. 동적인 웹 앱을 만들 때는 React가 적합합니다.

중요한 것은 모든 것을 React로 만들 필요가 없다는 점입니다. 필요한 부분만 선별적으로 전환하는 것이 효율적입니다. SSALWorks 방법론에서 P3(프로토타입)을 먼저 만들고 S2(개발)에서 전환하는 이유가 바로 이것입니다.

상황에 맞는 도구를 선택하는 것이 좋은 개발자의 자질입니다. 처음부터 완벽하게 선택하기 어렵다면, 일단 HTML로 시작하고 필요할 때 전환하는 전략이 안전합니다.
