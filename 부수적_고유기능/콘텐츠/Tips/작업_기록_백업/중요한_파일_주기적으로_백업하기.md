# 3-2-1 원칙으로 중요한 프로젝트 파일 안전하게 백업하기

> 이 문서는 Git과 별개로 중요한 프로젝트 파일을 안전하게 백업하는 방법과 3-2-1 백업 원칙을 설명합니다.

---

## 왜 별도 백업이 필요한가

Git은 코드 버전 관리 도구이지 백업 도구가 아닙니다. Git 저장소가 손상되거나 로컬 PC에 문제가 생기면 작업물을 잃을 수 있습니다. 별도 백업으로 이중 안전장치를 마련해야 합니다.

---

## 3-2-1 백업 원칙

| 숫자 | 의미 |
|------|------|
| 3 | 최소 3개의 복사본 |
| 2 | 2개 이상의 서로 다른 매체 |
| 1 | 1개는 물리적으로 다른 장소 |

```
복사본 1: 로컬 PC (작업용)
복사본 2: 클라우드 (Google Drive, OneDrive)
복사본 3: 외장 HDD (집 외부 보관)
```

---

## 필수 백업 대상

| 파일/폴더 | 중요도 | 이유 |
|----------|:------:|------|
| `.claude/` | 매우 높음 | 작업 기록, 설정 |
| `CLAUDE.md` | 매우 높음 | 프로젝트 규칙 |
| `Production/` | 매우 높음 | 배포 코드 |
| `*.sql` 파일 | 높음 | DB 스키마 |
| `S0_*/ssal-grid/` | 높음 | Grid 데이터 |

---

## 백업 불필요 대상

```
node_modules/     → npm install로 복구 가능
dist/, build/     → 빌드 명령으로 재생성
.next/, .cache/   → 캐시 파일
*.log             → 로그 파일
```

---

## 백업 방법 비교

| 방법 | 장점 | 단점 |
|------|------|------|
| 클라우드 동기화 | 자동, 실시간 | 용량 제한, 동기화 충돌 |
| 외부 드라이브 | 랜섬웨어 안전 | 수동 작업 필요 |
| NAS | 대용량, 자동화 | 초기 비용, 설정 복잡 |
| GitHub | 무료, 협업 용이 | 민감정보 주의 |

---

## 권장 백업 주기

| 주기 | 대상 | 방법 |
|------|------|------|
| 실시간 | 작업 중인 파일 | 클라우드 동기화 |
| 매일 | 코드, 작업 로그 | Git commit + push |
| 매주 | 전체 프로젝트 | 외장 드라이브 복사 |
| 매월 | 전체 백업 검증 | 복구 테스트 |

---

## Claude Code에게 요청하기

```
"현재 프로젝트 중요 파일 목록 정리해줘"
"백업해야 할 파일 찾아줘"
".gitignore에 백업 불필요 파일 추가해줘"
```
