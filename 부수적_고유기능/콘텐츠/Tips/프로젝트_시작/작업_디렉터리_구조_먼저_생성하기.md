# 작업 디렉터리 구조 먼저 생성하기

## 핵심 요약

웹 개발 프로젝트 시작 시 가장 먼저 해야 할 일은 로컬 파일에 프로젝트용 작업 디렉터리 구조를 생성하는 것입니다. 체계적인 폴더 구조가 있어야 파일 관리가 쉬워집니다.

## 왜 디렉터리 구조를 먼저 만들어야 하나?

### 1. 코드부터 작성하면 생기는 문제

```
❌ 잘못된 순서:
코드 작성 → 파일 여기저기 저장 → 나중에 정리 → 혼란 가중
```

- 파일이 루트에 쌓임
- 비슷한 파일이 여러 곳에 분산
- 나중에 정리하려면 시간이 배로 소요

### 2. 구조부터 만들면 좋은 점

```
✅ 올바른 순서:
폴더 구조 설계 → 문서화 → 코드 작성 → 제자리에 저장
```

- 처음부터 깔끔한 구조 유지
- 파일 저장 위치 고민 불필요
- 협업 시 혼란 방지

## 권장 폴더 구조 (단계별 + 영역별 융합)

프로젝트를 **단계(Stage)**와 **영역(Area)**으로 구분하여 체계적으로 관리합니다.

```
my-project/
│
# ========== 특별단계 (프로젝트 기초) ==========
├── P0_프로젝트_구조/                    # 프로젝트 구조 및 상태 관리
│   ├── Project_Directory_Structure.md   # 디렉토리 구조 문서
│   └── Project_Status.md                # 프로젝트 진행 상황
│
# ========== 예비단계 (기획 및 설계) ==========
├── P1_사업계획/                         # 비즈니스 계획
│   ├── Vision_Mission/
│   ├── Market_Analysis/
│   └── Business_Model/
├── P2_프로젝트_기획/                    # 프로젝트 기획
│   ├── Requirements/                    # 요구사항
│   ├── User_Flows/                      # 사용자 플로우
│   ├── Design_System/                   # 디자인 시스템
│   └── Tech_Stack/                      # 기술 스택
├── P3_프로토타입/                       # 프로토타입 (DB 연결)
│   ├── Frontend/                        # 프론트엔드 프로토타입
│   ├── Database/                        # DB 스키마
│   └── Documentation/                   # 문서화
│
# ========== 실행단계 (개발 및 운영) ==========
├── S1_개발_준비/                        # 개발 환경 설정
│   ├── Backend_Infra/                   # 백엔드 인프라
│   ├── Database/                        # DB 초기 설정
│   ├── Security/                        # 보안 설정
│   └── Testing/                         # 테스트 환경
├── S2_개발_1차/                         # 핵심 기능 개발
│   ├── Frontend/                        # 프론트엔드
│   ├── Backend_API/                     # 백엔드 API
│   ├── Database/                        # DB 스키마
│   └── Testing/                         # 테스트
├── S3_개발_2차/                         # 확장 기능 개발
│   ├── Frontend/
│   ├── Backend_API/
│   └── External/                        # 외부 서비스 연동
├── S4_개발_3차/                         # 고급 기능 및 QA
│   ├── Frontend/
│   ├── Backend_API/
│   └── Testing/
├── S5_운영/                             # 배포 및 운영
│   ├── DevOps/                          # CI/CD, 모니터링
│   └── Documentation/                   # 운영 문서
│
# ========== 배포용 코드 ==========
├── Production/                          # 실제 배포 코드 (최신 상태)
│   ├── Frontend/
│   ├── Backend_API/
│   └── Database/
│
# ========== 설정 폴더 ==========
├── .claude/                             # Claude Code 설정
│   ├── CLAUDE.md                        # 작업 규칙
│   └── work_logs/                       # 작업 로그
│       └── current.md
├── .git/                                # Git 저장소
│
# ========== 루트 파일 ==========
├── .gitignore
└── package.json
```

## 구조 설계 핵심 원칙

### 1. 단계별 분리 (P1 → P2 → P3 → S1 → S2 → ...)

- **P 단계 (Preparation)**: 기획, 설계, 프로토타입
- **S 단계 (Stage)**: 실제 개발 및 운영
- 순서대로 진행하며 각 단계의 산출물이 다음 단계의 입력

### 2. 영역별 하위 폴더

각 단계 안에서 영역(Area)별로 분류:

| 영역 | 폴더명 | 내용 |
|------|--------|------|
| 프론트엔드 | Frontend/ | HTML, CSS, JS |
| 백엔드 API | Backend_API/ | API 코드 |
| 데이터베이스 | Database/ | SQL 스키마 |
| 보안 | Security/ | 인증, 권한 |
| 테스트 | Testing/ | 테스트 코드 |
| 인프라 | Backend_Infra/ | 서버 설정 |
| 문서 | Documentation/ | 문서화 |

### 3. Production 폴더 = 배포 기준점

- 각 Stage에서 완료된 코드는 Production/에 반영
- Production/은 항상 배포 가능한 최신 상태 유지
- "이 폴더가 실제 서비스"라는 명확한 기준

## 실전 구현 방법

### Claude Code에게 요청

```
"프로젝트를 시작합니다. 다음 구조로 폴더를 생성해주세요:
- P0_프로젝트_구조/ (Project_Directory_Structure.md, Project_Status.md)
- P1_기획/
- P2_설계/
- S1_개발_준비/
- S2_개발_1차/
- Production/
- .claude/ (CLAUDE.md, work_logs/)
그리고 Project_Directory_Structure.md에 전체 구조를 문서화해주세요."
```

## 주의사항

- **폴더 구조를 먼저 만들고** 코딩 시작
- 빈 폴더도 미리 생성 (.gitkeep으로 Git 추적)
- 구조 변경 시 문서도 함께 업데이트
- **새 폴더 생성은 신중하게** (기존 구조에서 적합한 곳 먼저 찾기)
