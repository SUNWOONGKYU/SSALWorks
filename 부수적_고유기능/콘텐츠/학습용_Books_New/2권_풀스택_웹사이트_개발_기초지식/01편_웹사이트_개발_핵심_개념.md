# 1편 | 웹사이트 개발 핵심 개념

---

웹사이트 개발의 세계에 발을 들이면 정말 다양한 용어들이 쏟아져 나와요. 홈페이지, 웹사이트, 플랫폼, 프론트엔드, 백엔드... 처음엔 뭔가 복잡해 보이지만 걱정하지 마세요. 이번 편에서는 이 핵심 개념들을 **일상적인 비유**로 쉽게 풀어드릴게요. 마치 친한 선배가 옆에서 차근차근 설명해주는 것처럼 말입니다. 홈페이지와 웹사이트가 어떻게 다른지, 프론트엔드와 백엔드가 어떤 역할을 하는지 하나씩 알아가다 보면, 웹사이트 개발이 훨씬 친근하게 느껴질 것입니다.

## 1. 홈페이지 vs 웹사이트 vs 플랫폼 차이점


"홈페이지 하나 만들어주세요" 이런 말, 정말 자주 들어봤을 것입니다. 하지만 실제로는 웹사이트 전체를 원하는 경우가 대부분입니다. 이런 미묘한 차이를 정확히 이해하지 못하면, 나중에 "어? 이건 원한 게 아닌데..." 하는 상황이 벌어질 수 있습니다. 그래서 처음부터 개념을 명확히 잡아두는 것이 현명합니다.

### 1-1. 홈페이지 (Homepage)

**정의**: 웹사이트의 첫 페이지, 메인 페이지라고 생각하면 돼요

**쉬운 설명**: 집의 현관문을 생각해보면, 손님이 처음 들어와서 마주하는 공간이죠

**특징**:
- 웹사이트에 들어가면 제일 먼저 보는 화면입니다
- 회사 소개와 핵심 서비스를 한눈에 보여주죠
- 다른 페이지로 이동하는 **관문 역할**을 해요

**예시**: Samsung.com에 들어가보세요. 제일 먼저 보이는 그 화면이 바로 홈페이지입니다.

### 1-2. 웹페이지 (Webpage)

**정의**: 브라우저에서 볼 수 있는 하나하나의 페이지라고 생각하면 돼요

**쉬운 설명**: 두꺼운 책의 한 페이지를 생각해보면, 딱 그런 개념입니다.

**특징**:
- 하나의 HTML(하이퍼텍스트 마크업 언어; HyperText Markup Language) 파일이 하나의 웹페이지가 됩니다
- 홈페이지, 회사소개 페이지, 연락처 페이지 등이 모두 각각의 웹페이지입니다
- 여러 웹페이지들이 모여서 하나의 웹사이트를 구성합니다

### 1-3. 웹사이트 (Website)

**정의**: 여러 웹페이지들이 모여 있는 전체 사이트라고 보면 돼요

**쉬운 설명**: 집 전체를 떠올려보세요. 현관, 거실, 방, 화장실... 이 모든 공간이 합쳐진 것처럼 말입니다

**특징**:
- 홈페이지 + 회사소개 + 서비스소개 + 문의 등 **여러 페이지가 하나로 연결**되어 있어요
- 주로 정보 전달이 핵심 목적이죠
- 일방향 소통 형태예요 (회사 → 고객으로만)

**예시**: Samsung.com 전체가 웹사이트입니다 (홈페이지, 제품 페이지, 지원 페이지, 회사소개 등 모든 페이지 포함)

### 1-4. 플랫폼 (Platform)

**정의**: 사용자들이 서로 소통하고 거래할 수 있는 **살아있는 디지털 공간**이라고 생각해보세요

**쉬운 설명**: 붐비는 백화점이나 전통시장을 생각해보면, 여러 업체와 고객들이 만나고, 대화하고, 거래하는 생생한 장소 말입니다

**특징**:
- **양방향/다방향 소통**이 활발하게 일어나요
- 사용자 간 거래나 교환이 자연스럽게 이루어져요
- 하나의 완전한 **생태계**를 만들어내죠

**예시**: 쿠팡(판매자와 구매자가 만나는 공간), 유튜브(창작자와 시청자가 만나는 공간)

### 1-5. 랜딩 페이지 (Landing Page)

**정의**: **하나의 명확한 목적**을 위해 특별히 만든 독립적인 웹페이지라고 생각하면 돼요

**쉬운 설명**: 비행기가 착륙(Landing)하는 활주로를 떠올려보세요. 방문자가 광고를 클릭하고 '착륙'하는 그 첫 번째 페이지가 바로 랜딩 페이지입니다

**특징**:
- **하나의 명확한 목표**만 가져요 (회원가입, 구매, 다운로드 등)
- 광고나 검색을 통해 처음 '착륙'하는 페이지죠
- 홈페이지와는 달리 오직 **전환(구매/가입)에만 집중**해요

**예시**: "무료 전자책 다운로드" 광고를 클릭했을 때 나오는 다운로드 전용 페이지

### 1-6. 개발 계획 수립 시 필수 고려사항

**미래 확장성을 고려한 기획의 중요성**

"일단 홈페이지부터 만들어보자" 많은 분들이 이렇게 생각합니다. 하지만 **주의해야 할 점은** 이런 접근이 나중에 큰 함정이 될 수 있다는 것입니다. 마치 작은 원룸에 살다가 갑자기 대가족이 되었는데, 집을 통째로 다시 지어야 하는 상황 같아요. 처음부터 최종 목표를 명확히 하고 단계적으로 접근하는 것이 **훨씬 현명하고 경제적**입니다.

**올바른 접근 방법:**
1. **최종 목표 설정**: 홈페이지로 끝낼 것인가? 웹사이트까지? 플랫폼까지?
2. **확장 가능한 구조로 설계**: 나중에 기능을 추가할 수 있도록 기반을 마련
3. **단계적 개발**: 홈페이지 → 웹사이트 → 플랫폼으로 점진적 확장

## 2. 프론트엔드와 백엔드 역할 구분


웹사이트를 레스토랑으로 비유하면 프론트엔드와 백엔드의 역할을 쉽게 이해할 수 있습니다.

### 2-1. 프론트엔드 (Frontend)

**쉬운 설명**: 사용자가 보고 클릭하는 웹사이트의 겉모습입니다

**레스토랑 비유**: 홀, 인테리어, 메뉴판, 웨이터 서비스에 해당합니다

**역할**: 예쁘고 사용하기 편한 화면을 만듭니다

**구체적인 일:**
- 버튼을 디자인하고 색깔을 정합니다
- 메뉴를 배치하고 폰트를 선택합니다
- 클릭했을 때 반응하는 애니메이션을 만듭니다
- 스마트폰에서도 예쁘게 보이도록 조정합니다

**사용 기술**: HTML(하이퍼텍스트 마크업 언어), CSS(종속형 시트), JavaScript(자바스크립트), React 등을 활용합니다

### 2-2. 백엔드 (Backend)

**쉬운 설명**: 사용자가 보지 못하는 서버에서 돌아가는 프로그램입니다

**레스토랑 비유**: 주방, 요리사, 재료 창고, 계산 시스템에 해당합니다

**역할**: 실제 비즈니스 로직을 처리합니다

**구체적인 일:**
- 서비스 요청 신청서가 제출되면 이메일로 알림을 보냅니다
- 로그인 정보를 확인합니다
- 데이터베이스에서 필요한 정보를 가져옵니다
- 결제를 처리하고 영수증을 발행합니다

**사용 기술**: Node.js, Python, Java, MySQL 등을 활용합니다

### 2-3. 상호 협력 관계

고객이 "서비스 신청" 버튼을 클릭하면:
1. **프론트엔드**: 예쁜 신청 폼을 보여줌
2. **백엔드**: 신청 정보를 받아서 데이터베이스에 저장
3. **프론트엔드**: "신청이 완료되었습니다" 메시지 표시

## 3. 웹사이트 개발에서 자주 쓰는 필수 용어들

### 3-1. 기술 관련 용어

**브라우저 (Browser)**
- 웹사이트를 보여주는 프로그램 (크롬, 사파리, 엣지 등)
- 인터넷 주소를 입력하면 웹페이지를 화면에 표시해주는 도구
- 웹사이트와 사용자 사이의 창문 역할

**HTML(하이퍼텍스트 마크업 언어; HyperText Markup Language)**
- 웹페이지의 뼈대를 만드는 언어
- 제목, 문단, 이미지, 링크 등의 구조 정의

**CSS(종속형 시트; Cascading Style Sheets)**
- 웹페이지를 예쁘게 꾸미는 언어
- 색깔, 크기, 위치, 애니메이션 등 디자인 담당

**JavaScript(자바스크립트)**
- 웹페이지에 동작을 추가하는 언어
- 버튼 클릭, 메뉴 열기, 계산기 기능 등

**데이터베이스 (Database)**
- 정보를 체계적으로 저장하는 디지털 창고
- 고객 정보, 상품 정보, 주문 내역 등 보관

**서버 (Server)**
- 웹사이트 파일들을 저장하고 방문자에게 전송하는 컴퓨터
- 24시간 켜져있어서 언제든 접속 가능

**클라우드 (Cloud)**
- 내 컴퓨터가 아닌 인터넷 상의 서버 컴퓨터를 빌려서 사용하는 것
- 예시: 구글 드라이브, 네이버 클라우드, AWS(아마존 웹 서비스; Amazon Web Services)
- 장점: 언제 어디서나 접근, 백업 걱정 없음

**도메인 (Domain)**
- 웹사이트의 주소 이름 (예: google.com, naver.com)
- 인터넷 상에서 웹사이트를 찾아가는 이름표

**호스팅 (Hosting)**
- 웹사이트 파일들을 저장해주는 서비스
- 인터넷에 연결된 컴퓨터에 내 웹사이트를 올려두는 것

### 3-2. 개발 프로세스 용어

**프레임워크 (Framework)**
- 개발을 쉽게 해주는 미리 만들어진 도구 세트
- 집 짓기에서 건축용 크레인 같은 역할

**라이브러리 (Library)**
- 특정 기능을 위한 미리 만들어진 코드 모음
- 집 짓기에서 미리 만들어진 문이나 창문 같은 것

**API(응용 프로그램 인터페이스; Application Programming Interface)**
- 서로 다른 프로그램끼리 대화하는 방법
- 식당의 웨이터 (주방과 손님 사이 메시지 전달)

**Git (깃)**
- 코드 버전 관리 시스템
- 여러 개발자가 함께 작업할 때 코드 변경사항을 추적
- 이전 버전으로 되돌리기, 여러 버전 병합하기 가능
- GitHub(깃허브), GitLab(깃랩) 등의 서비스와 함께 사용

**배포 (Deploy)**
- 개발한 웹사이트를 실제 서버에 올려서 공개하는 과정
- 로컬 개발 환경에서 실제 운영 환경으로 이동
- 자동 배포 시스템을 통해 코드 수정 시 자동으로 반영 가능

### 3-3. 웹사이트 구조 및 디자인 용어

**UI/UX(사용자 인터페이스/사용자 경험; User Interface / User Experience)**
- UI(사용자 인터페이스): 사용자가 보고 조작하는 화면 인터페이스 (버튼, 메뉴, 색상 등)
- UX(사용자 경험): 사용자가 웹사이트를 이용하면서 느끼는 전체적인 경험
- 좋은 UI(사용자 인터페이스): 예쁘고 직관적인 디자인
- 좋은 UX(사용자 경험): 원하는 정보를 쉽고 빠르게 찾을 수 있는 구조

**CMS(콘텐츠 관리 시스템; Content Management System)**
- 웹사이트 내용을 쉽게 관리할 수 있는 시스템
- 코딩 지식 없이도 글쓰기, 이미지 업로드, 페이지 수정 가능
- 대표적인 예: 워드프레스(WordPress), 네이버 블로그, 티스토리

**SPA(단일 페이지 애플리케이션; Single Page Application)**
- 페이지를 새로 불러오지 않고 내용만 바뀌는 웹사이트
- 예시: 네이버 메일처럼 메일함→보낸편지함 이동할 때 페이지가 새로고침 되지 않고 내용만 바뀌는 것
- 장점: 빠르고 부드러운 사용자 경험
- 전통적 방식과 차이: 전통적 방식은 페이지 전체를 다시 로드, SPA(단일 페이지 애플리케이션)는 내용만 변경

**반응형 디자인 (Responsive Design)**
- 스마트폰, 태블릿, PC 화면 크기에 맞춰 자동으로 레이아웃이 바뀌는 것
- 예시: PC에서는 1줄로 보이던 내용이 스마트폰에서는 3줄로 세로 정렬
- 스마트폰 사용자가 전체의 60% 이상이므로 필수
- 한 번 만들면 모든 기기에서 사용 가능

### 3-4. 보안 용어

**HTTPS(보안 하이퍼텍스트 전송 프로토콜; HyperText Transfer Protocol Secure)**
- 웹사이트와 사용자 간의 안전한 통신을 보장하는 프로토콜
- 주소창에 자물쇠 표시가 나타나는 보안 웹사이트
- 개인정보, 결제정보 등을 암호화해서 전송

**SSL/TLS(보안 소켓 계층/전송 계층 보안; Secure Sockets Layer/Transport Layer Security) 인증서**
- 웹사이트의 보안 연결을 보장하는 디지털 인증서
- HTTPS(보안 하이퍼텍스트 전송 프로토콜) 프로토콜을 사용하기 위해 필수
- 데이터 암호화와 웹사이트 신원 확인 기능

### 3-5. 성능 최적화 용어

**캐시 (Cache)**
- 자주 사용하는 데이터를 임시로 저장해두는 기술
- 웹사이트 로딩 속도를 빠르게 만들어줌
- 브라우저 캐시: 이미지, CSS(종속형 시트), JavaScript(자바스크립트) 파일 등을 로컬에 저장
- 서버 캐시: 데이터베이스 조회 결과 등을 서버에 임시 저장

**CDN(콘텐츠 전송 네트워크; Content Delivery Network)**
- 전 세계에 분산된 서버를 통해 콘텐츠를 빠르게 전달하는 시스템
- 사용자와 가장 가까운 서버에서 파일을 제공
- 대용량 이미지, 동영상 등을 효율적으로 전송
