# 13편 | 백엔드 기초와 Node.js

---

**"내가 만든 웹사이트가 실제 웹사이트가 되려면 무엇이 더 필요할까"라는 질문은 누구나 한 번쯤 해봅니다.** 지금까지 예쁜 화면을 만드는 프론트엔드와 편리한 개발 도구들을 알아보았는데, 이제 중요한 질문이 남았습니다.

**사용자가 회원가입을 하면 그 정보를 어딘가에 저장해야 합니다. 로그인 정보도 확인할 수 있어야 하고, 게시글도 어딘가에 보관되어야 합니다. 이런 문제들이 바로 백엔드가 해결하는 영역입니다.**

**마치 카페로 비유하면 다음과 같습니다.** 지금까지는 예쁜 인테리어와 좋은 의자, 멋진 메뉴판을 준비했다면, 이제는 **실제로 커피를 만들 주방과 재료 창고, 그리고 주문을 받는 시스템**이 필요합니다.

**바로 이것이 백엔드의 영역입니다.** 사용자는 절대 볼 수 없지만, **웹사이트의 심장과 두뇌 역할**을 하는 곳입니다. 그리고 **JavaScript만 알아도 백엔드를 만들 수 있는** Node.js라는 훌륭한 도구가 있습니다.

## 1. 백엔드의 역할과 필요성

### 1-1 백엔드란 무엇인가

**백엔드를 가장 쉽게 이해하는 방법은 백화점에 비유하는 것입니다.**

**프론트엔드 = 고객이 보는 매장 공간**
- 예쁜 진열대, 친절한 직원, 깔끔한 인테리어
- 고객이 직접 보고 터치하고 경험하는 모든 것

**백엔드 = 고객이 볼 수 없는 뒷공간**
- 창고, 사무실, 물류센터, 계산 시스템
- 고객은 모르지만 **매장이 돌아가게 하는 모든 핵심 시스템**

**실제 예시로 설명해보겠습니다:**

**네이버에 로그인할 때 일어나는 일들:**
1. 여러분이 아이디/비밀번호 입력 (프론트엔드)
2. **백엔드가 "이 사람이 정말 맞는지" 확인하며 데이터베이스 검토**
3. **백엔드가 "응, 맞습니다" 하며 로그인 허가**
4. 메인 화면 표시 (프론트엔드)

**핵심은 다음과 같습니다**: 사용자는 2~3번 과정을 전혀 볼 수 없지만, **이 과정이 없으면 웹사이트가 작동하지 않습니다.** 마치 **백화점의 숨은 영웅들**처럼 백엔드는 보이지 않는 곳에서 모든 것을 책임지고 있습니다.

### 1-2 프론트엔드와 백엔드의 차이

프론트엔드는 브라우저에서 실행되는 코드입니다. HTML, CSS, JavaScript로 만들어지며, 사용자가 직접 보고 상호작용하는 부분입니다. 누구나 브라우저의 개발자 도구(F12)를 열면 프론트엔드 코드를 볼 수 있습니다.

백엔드는 서버에서 실행되는 코드입니다. 사용자는 백엔드 코드를 직접 볼 수 없습니다. API를 통해서만 백엔드와 통신합니다. API는 프론트엔드와 백엔드가 데이터를 주고받는 규칙입니다. 비밀번호 확인, 결제 처리, 데이터베이스 관리 같은 중요한 작업들이 백엔드에서 안전하게 처리됩니다.

### 1-3 백엔드가 필요한 이유

프론트엔드만으로는 할 수 없는 일들이 많습니다. 첫째, 데이터를 영구적으로 저장할 수 없습니다. 브라우저를 닫으면 모든 데이터가 사라집니다. 둘째, 보안이 취약합니다. 프론트엔드 코드는 누구나 볼 수 있어서 중요한 정보를 처리할 수 없습니다. 셋째, 여러 사용자 간의 데이터 공유가 불가능합니다.

온라인 쇼핑몰을 예로 들면, 상품 정보는 모든 사용자가 같은 데이터를 봐야 하고, 재고는 실시간으로 관리되어야 하며, 결제 정보는 안전하게 처리되어야 합니다. 이 모든 것이 백엔드 없이는 불가능합니다.

### 1-4 백엔드 개발 언어들

백엔드는 다양한 프로그래밍 언어로 개발할 수 있습니다. Java(Spring), Python(Django, Flask), Ruby(Rails), PHP(Laravel), C#(.NET) 등이 전통적으로 많이 사용되었습니다. 각 언어마다 장단점이 있고, 기업이나 프로젝트의 특성에 따라 선택됩니다.

Node.js는 JavaScript로 백엔드를 개발할 수 있게 해주는 런타임입니다. 런타임은 JavaScript 코드를 브라우저 밖에서 실행할 수 있게 해주는 환경입니다. 프론트엔드와 같은 언어를 사용할 수 있어서 풀스택 개발이 쉽습니다. 풀스택은 프론트엔드와 백엔드를 모두 다루는 것을 말합니다. npm 생태계를 그대로 활용할 수 있다는 장점도 있습니다.

## 2. Node.js의 이해

### 2-1 Node.js가 생긴 이유

**시간을 2009년으로 돌려보겠습니다.** 그때는 JavaScript가 "브라우저에서만 쓸 수 있는 언어"였습니다. 마치 수영장에서만 수영할 수 있는 상황이었습니다.

**그런데 Ryan Dahl이라는 개발자가 이런 생각을 했습니다:** "Chrome 브라우저의 V8 엔진이 이렇게 빠른데, 이것을 브라우저 밖에서도 사용할 수 있다면 좋을 것 같다."

**그래서 탄생한 것이 바로 Node.js입니다.** 마치 수영을 수영장에서만 하던 사람이 갑자기 **바다에서도, 강에서도, 심지어 우주에서도** 수영할 수 있게 된 것입니다.

**Node.js가 가져다준 혁신적인 변화들:**
- **서버 개발**: JavaScript로 서버를 만들 수 있다는 것이 혁신적입니다.
- **데스크톱 앱**: 슬랙, 디스코드 같은 앱들도 JavaScript로 개발 가능합니다.
- **IoT 기기**: 심지어 스마트 전구도 제어 가능합니다.

**그 결과로 나타난 놀라운 성과:** 프론트엔드 개발자가 백엔드도 개발할 수 있게 되었고, **Netflix, PayPal, LinkedIn** 같은 거대 기업들도 Node.js를 사용하고 있습니다.

### 2-2 Node.js의 특징

```
[백엔드 아키텍처]

┌─────────────────────────────────────────┐
│           Client (브라우저)              │
└─────────────────────────────────────────┘
                    ↓↑
                  HTTPS
                    ↓↑
┌─────────────────────────────────────────┐
│           Load Balancer                 │
│         (트래픽 분산)                   │
└─────────────────────────────────────────┘
                    ↓↑
    ┌───────────────┼───────────────┐
    ↓               ↓               ↓
┌─────────┐   ┌─────────┐   ┌─────────┐
│ Server1 │   │ Server2 │   │ Server3 │
│(Node.js)│   │(Node.js)│   │(Node.js)│
└─────────┘   └─────────┘   └─────────┘
      ↓             ↓             ↓
      └─────────────┼─────────────┘
                    ↓
            ┌──────────────┐
            │   Database   │
            │   (MySQL)    │
            └──────────────┘
```

**Node.js를 특별하게 만드는 것은 바로 '비동기 처리' 능력입니다.** 이를 카페 알바생으로 비유해보겠습니다.

**일반적인 서버 (동기 처리):**
- 커피 주문 1번 받기 → 커피 만들기 → 서빙하기 → 그다음 주문 2번 받기
- **한 번에 하나씩만 처리**
- 매우 느립니다.

**Node.js 서버 (비동기 처리):**
- 커피 주문 1번 받고 → **바로** 주문 2번, 3번, 4번도 쭉쭉 받기
- 커피가 만들어지는 동안 **다른 일들을 계속 처리**
- **마치 8개 팔을 가진 알바생**

**또 다른 특징: 싱글 스레드**
- **"한 명의 직원이지만 엄청 빠르게 일처리하는 방식"**
- 메모리를 적게 사용해서 효율적
- **가벼운 작업을 많이 처리**하는 데 최적화

**그래서 Node.js가 특히 잘하는 일들:**
- **실시간 채팅**: 메시지가 빠르게 전달됩니다.
- **API 서버**: 수많은 요청을 동시에 처리
- **소셜미디어**: 동시 접속자가 많아도 문제없음

**한마디로 요약하면: "적은 자원으로 많은 일을 동시에 처리"입니다.**

### 2-3 Node.js 설치와 실행

Node.js는 nodejs.org에서 다운로드할 수 있습니다. LTS(Long Term Support) 버전을 선택하는 것이 안정적입니다. 2025년 9월 현재 Node.js 22가 Active LTS 버전입니다 (2024년 10월부터). 설치가 완료되면 명령줄(CLI)에서 `node -v`로 버전을 확인할 수 있습니다.

```bash
node -v  # Node.js 버전 확인
npm -v   # npm 버전 확인
node app.js  # JavaScript 파일 실행
```

Node.js를 설치하면 npm도 자동으로 설치됩니다. npm을 통해 수십만 개의 패키지를 사용할 수 있어서, 필요한 기능을 직접 만들지 않고 가져다 쓸 수 있습니다.

### 2-4 Node.js의 모듈 시스템

Node.js는 기능을 모듈로 나누어 관리합니다. 내장 모듈(fs, http, path 등)은 별도 설치 없이 사용할 수 있고, 외부 모듈은 npm으로 설치해서 사용합니다.

```javascript
// 내장 모듈 사용
const http = require('http');
const fs = require('fs');
const path = require('path');

// 외부 모듈 사용 (npm install express 후)
const express = require('express');
```

자신만의 모듈을 만들어서 재사용할 수도 있습니다. 관련 기능을 하나의 파일에 모아서 exports로 내보내고, 다른 파일에서 require로 가져와서 사용합니다.

## 3. Express 프레임워크

### 3-1 Express란 무엇인가

Express는 Node.js를 위한 가장 인기 있는 웹 프레임워크입니다. 프레임워크는 개발을 쉽고 빠르게 할 수 있도록 기본 구조와 기능을 제공하는 도구 모음입니다. Node.js의 기본 http 모듈만으로도 서버를 만들 수 있지만, Express를 사용하면 훨씬 간단하고 강력한 서버를 만들 수 있습니다. 마치 자전거와 자동차의 차이처럼, Express는 더 많은 기능과 편의성을 제공합니다.

```javascript
// Express로 간단한 서버 만들기
const express = require('express');
const app = express();

app.get('/', (req, res) => {
    res.send('안녕하세요!');
});

app.listen(3000, () => {
    console.log('서버가 3000번 포트에서 실행 중');
});
```

### 3-2 Express의 라우팅 시스템

라우팅은 URL 경로에 따라 다른 응답을 보내는 것입니다. 사용자가 /home으로 접속하면 홈페이지를, /about으로 접속하면 소개 페이지를 보여주는 식입니다.

```javascript
app.get('/', (req, res) => {
    res.send('홈페이지');
});

app.get('/about', (req, res) => {
    res.send('소개 페이지');
});

app.post('/login', (req, res) => {
    // 로그인 처리
    res.send('로그인 성공');
});
```

GET은 데이터를 조회할 때, POST는 데이터를 생성할 때 사용합니다. PUT은 수정할 때, DELETE는 삭제할 때 사용합니다. RESTful API의 원칙을 따라 직관적인 URL 구조를 만들 수 있습니다. RESTful API는 URL과 HTTP 메서드를 조합해 자원을 다루는 표준화된 방식입니다.

### 3-3 Express의 미들웨어

미들웨어는 요청과 응답 사이에서 실행되는 함수입니다. 함수는 특정 작업을 수행하는 코드 묶음입니다. 공장의 조립 라인처럼, 요청이 들어오면 여러 미들웨어를 거쳐가며 처리됩니다. 인증 확인, 로그 기록, 데이터 파싱 등의 작업을 미들웨어로 처리합니다. 데이터 파싱은 받은 데이터를 프로그램이 이해할 수 있는 형태로 변환하는 것입니다.

```javascript
// 모든 요청에 대해 로그 기록
app.use((req, res, next) => {
    console.log(`${req.method} ${req.url}`);
    next();  // 다음 미들웨어로 전달
});

// JSON 데이터 파싱
app.use(express.json());

// 정적 파일 제공
app.use(express.static('public'));
```

### 3-4 Express와 템플릿 엔진

템플릿 엔진은 동적으로 HTML을 생성하는 도구입니다. 동적이란 데이터에 따라 내용이 바뀌는 것을 말합니다. 데이터베이스에서 가져온 정보를 HTML에 삽입해서 각 사용자마다 다른 페이지를 보여줄 수 있습니다. EJS, Pug, Handlebars 등이 많이 사용됩니다.

```javascript
// EJS 템플릿 엔진 설정
app.set('view engine', 'ejs');

app.get('/user/:id', (req, res) => {
    // 실제로는 데이터베이스에서 사용자 정보를 가져옴
    const userData = {
        name: '김철수',
        email: 'user@example.com'
    };
    res.render('profile', { user: userData });
});
```

템플릿 엔진을 사용하면 HTML 파일에서 <%= %> 같은 특수 문법으로 변수를 삽입할 수 있습니다. 이런 HTML 파일을 템플릿 파일이라고 부릅니다. 반복문이나 조건문도 사용할 수 있어서 동적인 페이지를 쉽게 만들 수 있습니다.

## 4. 웹 페이지 생성 방식의 종류

지금까지 백엔드의 역할, Node.js, Express 프레임워크에 대해 알아보았습니다. 이제 이러한 백엔드 기술들이 실제로 웹 페이지를 만들어내는 다양한 방식들을 살펴보겠습니다. Node.js와 Express를 사용하면 아래의 모든 방식으로 웹 애플리케이션을 구현할 수 있습니다.

### 4-1 서버 사이드 렌더링 (SSR)

서버 사이드 렌더링은 서버에서 HTML을 완성해서 브라우저에 전송하는 방식입니다. 전통적인 웹사이트 방식입니다. 브라우저는 받은 HTML을 그대로 화면에 표시합니다. Express와 템플릿 엔진을 조합하면 이 방식으로 웹사이트를 만들 수 있습니다. EJS나 Pug가 대표적인 템플릿 엔진입니다.

SSR의 장점은 초기 로딩이 빠르고 SEO에 유리하다는 것입니다. SEO는 검색엔진 최적화를 의미합니다. 검색엔진이 완성된 HTML을 바로 읽을 수 있어서 구글이나 네이버에서 검색이 잘 됩니다. 단점은 페이지 이동 시마다 전체 페이지를 다시 로드해야 해서 사용자 경험이 떨어질 수 있습니다.

### 4-2 클라이언트 사이드 렌더링 (CSR)

클라이언트 사이드 렌더링은 브라우저에서 JavaScript로 HTML을 생성하는 방식입니다. 서버는 빈 HTML과 JavaScript 파일만 전송합니다. 브라우저에서 JavaScript가 실행되면서 화면을 그립니다. Express는 주로 API 서버 역할을 하며, React, Vue, Angular 같은 프론트엔드 프레임워크가 화면을 담당합니다.

CSR의 장점은 페이지 전환이 빠르고 사용자 경험이 좋다는 것입니다. 앱처럼 부드럽게 작동합니다. 단점은 초기 로딩이 느리고, JavaScript가 실행되기 전까지는 빈 화면만 보인다는 것입니다. SEO에도 불리할 수 있습니다.

### 4-3 하이브리드 렌더링

최근에는 SSR과 CSR의 장점을 합친 하이브리드 방식이 인기입니다. Next.js나 Nuxt.js 같은 프레임워크가 이를 지원합니다. Next.js는 React 기반이고 Nuxt.js는 Vue 기반입니다. 첫 페이지는 서버에서 렌더링하고, 이후 페이지 이동은 클라이언트에서 처리합니다.

이 방식을 사용하면 초기 로딩도 빠르고, SEO도 좋으며, 사용자 경험도 뛰어납니다. 다만 설정이 복잡하고 서버 비용이 더 들 수 있습니다.

### 4-4 정적 사이트 생성 (SSG)

정적 사이트 생성은 빌드 시점에 모든 페이지를 미리 만들어두는 방식입니다. 블로그나 문서 사이트처럼 자주 바뀌지 않는 콘텐츠에 적합합니다. Gatsby, Next.js, Hugo 같은 도구를 사용합니다.

SSG의 가장 큰 장점은 속도입니다. 이미 만들어진 HTML 파일을 그대로 전송하기 때문에 매우 빠릅니다. CDN에 배포하기도 쉽고 서버 비용도 거의 들지 않습니다. 단점은 데이터가 바뀔 때마다 다시 빌드해야 한다는 것입니다.
