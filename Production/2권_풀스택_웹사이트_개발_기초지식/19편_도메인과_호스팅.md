# 19편 | 웹사이트 배포 - 도메인과 호스팅

---

18편에서 보안과 성능 최적화를 알아보았는데, 이제는 **여러분이 만든 웹사이트를 전 세계에 공개**하는 단계입니다. 지구 반대편에 있는 누군가가 여러분의 웹사이트에 접속해서 "와, 이거 누가 만든 거냐!"라고 평가할 수도 있습니다. 배포가 무엇인지부터 시작해 도메인 이름의 작동 원리와 구매 방법, 그리고 다양한 호스팅 서비스 비교까지 **웹사이트를 인터넷이라는 바다에 띄우는 전 과정**을 살펴보겠습니다.

## 1. 배포란 무엇인가

### 1-1 배포의 개념

**배포(Deploy)**는 개발한 웹사이트를 인터넷에서 접속할 수 있도록 **세상에 공개하는 매우 중요한 과정**입니다. 지금까지 우리가 만든 HTML, CSS, JavaScript 파일들은 내 컴퓨터 안에만 조용히 숨어있어서 다른 사람들이 볼 수 없습니다. 하지만 이 파일들을 웹 서버에 올려놓는 순간, 전 세계 누구나 접근할 수 있게 됩니다.
배포를 책 출판에 비유하면 이해하기 쉬워집니다. 원고를 혼자 방 안에서 끙끙거리며 작성하는 것이 개발이라면, 그 원고를 인쇄해서 전국 서점에 진열하는 것이 바로 배포입니다. 아무리 대단한 소설을 써도 서점에 나가지 않으면 아무도 읽을 수 없듯이, **배포를 해야 비로소 다른 사람들이 내 작품을 발견하고 평가할 수 있게** 되는 것입니다.

### 1-2 배포에 필요한 요소

웹사이트를 배포하려면 다음 요소들이 필요합니다:

1. **호스팅 서버**: 웹사이트 파일을 저장할 컴퓨터
2. **도메인**: 웹사이트 주소 (선택사항이지만 거의 필수)
3. **배포 도구**: 파일을 서버로 전송하는 방법 (FTP, Git 등)
4. **SSL 인증서**: 보안 연결을 위한 인증서

### 1-3 배포 과정

일반적인 배포 과정은 다음과 같습니다:

1. **개발 완료**: HTML, CSS, JavaScript 파일 준비
2. **호스팅 선택**: 서버 공간 확보
3. **파일 업로드**: FTP나 Git을 통해 파일 전송
4. **도메인 연결**: 구매한 도메인을 서버와 연결
5. **SSL 설정**: HTTPS 보안 연결 설정
6. **테스트**: 실제 접속해서 확인

```
[배포 프로세스]

로컬 개발 → 빌드 → 테스트 → 통과?
                           ↓ Yes
                    서버 업로드 → 도메인 연결 → SSL 설정 → 배포 완료
                           ↓ No
                    로컬 개발로 돌아가서 수정
```

## 2. 도메인의 이해

**도메인**은 웹사이트의 주소를 사람이 기억하기 쉬운 문자로 표현한 것입니다. 복잡한 숫자로 이루어진 IP 주소를 대신해주는 역할을 합니다.

예를 들어 "강남역"이라는 이름이 GPS 좌표(위도 37.123, 경도 127.456)보다 기억하기 쉬운 것처럼, 도메인도 IP 주소보다 기억하기 쉬운 방식입니다.

google.com의 실제 IP 주소는 142.250.196.110 같은 숫자 조합입니다. 이런 숫자를 외우는 것보다 google.com이라는 도메인을 사용하는 것이 훨씬 효율적입니다.

### 2-1 도메인 구조의 이해

도메인은 계층 구조로 되어 있습니다.

**도메인 구조 분석:**
`example.com`을 기준으로 설명하면:

- **.com** : 최상위 도메인 (TLD - Top Level Domain)
- **example** : 두 번째 레벨 도메인
- **example.com** : 우리가 구매하는 도메인 (메인 도메인)


최상위 도메인의 종류는 다양합니다:
- **일반 최상위 도메인**: .com(상업용), .org(비영리단체), .net(네트워크), .edu(교육기관)
- **국가 코드 도메인**: .kr(한국), .jp(일본), .cn(중국), .us(미국)
- **신규 최상위 도메인**: .blog, .shop, .app, .dev, .io

### 2-2 서브도메인의 실제 활용

서브도메인은 하나의 도메인에서 여러 서비스를 구분할 때 사용합니다.

**네이버의 서브도메인 활용 사례:**
- **naver.com** = 네이버 메인 포털
- **mail.naver.com** = 네이버 메일 서비스
- **blog.naver.com** = 네이버 블로그 서비스
- **map.naver.com** = 네이버 지도 서비스
- **news.naver.com** = 네이버 뉴스 서비스


**핵심 정리:**
- 서브도메인은 메인 도메인 **앞에** 붙습니다
- 하나의 메인 도메인으로 여러 서브도메인을 무제한 생성 가능합니다
- 서브도메인마다 다른 서버나 다른 IP 주소를 지정할 수 있습니다

### 2-3 DNS의 작동 원리

**DNS**는 Domain Name System의 약자로, 도메인 이름을 IP 주소로 변환해주는 시스템입니다.

웹사이트에 접속하는 과정은 다음과 같습니다:

1. 브라우저에 google.com 입력
2. 컴퓨터가 DNS 서버에 IP 주소 요청
3. DNS 서버가 142.250.196.110 응답
4. 브라우저가 해당 IP 주소로 접속
5. 구글 서버가 웹페이지 전송

이 모든 과정은 몇 초 안에 완료됩니다.

DNS 서버는 계층적으로 동작합니다. 로컬 DNS 서버가 모르면 상위 DNS 서버에 물어보고, 최종적으로 해당 도메인을 관리하는 네임서버에서 정답을 찾습니다.

## 3. 호스팅의 이해

### 3-1 호스팅이란 무엇인가

**호스팅**은 웹사이트 파일을 저장하고 인터넷에 공개하는 **매우 중요한 서비스**입니다. 영어로 Hosting이라고 쓰는데, 쉽게 말해 **웹사이트가 살 집을 임대하는 것**과 같습니다. 여러분의 웹사이트도 어딘가에 살 곳이 필요합니다.

우리가 정성스럽게 만든 HTML, CSS, JavaScript 파일들은 지금 내 컴퓨터에 조용히 저장되어 있지만, 아쉽게도 다른 사람들이 접속할 수 없습니다. 하지만 이 파일들을 **24시간 365일 쉬지 않고 켜져 있는 서버 컴퓨터**에 올려놓는 순간, 전 세계 누구나 접속할 수 있게 됩니다. 이런 **서버 공간을 제공하는 매우 효과적인 서비스**가 바로 호스팅입니다.

호스팅 서비스가 제공하는 것들:
- 서버 공간 (파일 저장소)
- 도메인 연결 기능
- 이메일 서비스
- 데이터베이스
- 보안 기능 (SSL 인증서 등)
- 백업 서비스

### 3-2 호스팅의 종류

**공유 호스팅** - **"셰어하우스"** 방식
하나의 서버를 여러 사용자가 나눠 쓰는 방식입니다. Shared Hosting이라고도 하는데, 원룸을 여러 명이 나눠 쓰는 셰어하우스와 유사합니다.
- 장점: 저렴한 비용, 관리가 쉬움 (입문자에게 적합)
- 단점: 속도가 느릴 수 있음, 자원 제한 (다른 사람들과 나눠 써야 하니까)
- 적합한 경우: 개인 블로그, 소규모 웹사이트 (처음 시작하기에 적합)

**VPS 호스팅** - **"오피스텔"** 방식
하나의 서버를 가상으로 딱 나눠서 독립적인 공간을 제공하는 방식입니다. VPS는 Virtual Private Server의 약자이고, 오피스텔처럼 독립된 내만의 공간을 갖는다고 생각하면 됩니다.
- 장점: 독립적인 자원, 훨씬 나은 성능 (내가 배정된 자원을 독점)
- 단점: 공유 호스팅보다 비쌈, 어느 정도 기술 지식 필요 (어려울 수 있습니다)
- 적합한 경우: 중소규모 비즈니스 웹사이트 (사업이 어느 정도 커졌을 때)

**전용 서버 호스팅** - **"단독 주택"** 방식
서버 한 대를 완전히 내 것처럼 사용하는 방식입니다. Dedicated Hosting이라고도 하는데, 단독 주택을 전체 임대하는 것과 유사합니다.
- 장점: 최고의 성능, 완전한 제어권 (내가 왕이다)
- 단점: 매우 비쌈, 전문 지식 필수 (전문가가 아니면 힘든 선택)
- 적합한 경우: 대규모 웹사이트, 기업 서비스 (돈과 전문성이 필요합니다)

**클라우드 호스팅** - **"유연한 자원 활용"** 방식
여러 서버의 자원을 필요에 따라 **유연하고 스마트하게** 사용하는 최첨단 방식입니다. Cloud Hosting이라고도 합니다.
- 장점: 뛰어난 확장성, 최고의 안정성, 사용한 만큼만 비용 지불 (공정합니다)
- 단점: 비용 예측이 어려움 (가끔 예상보다 많이 나올 수도)
- 대표 서비스: AWS, Google Cloud, Microsoft Azure (거대 기업들의 선택)

### 3-3 정적 호스팅 서비스

최근에는 정적 웹사이트를 위한 전문 호스팅 서비스들이 인기입니다. HTML, CSS, JavaScript로만 이루어진 웹사이트를 쉽고 빠르게 배포할 수 있습니다.

**GitHub Pages**
GitHub 저장소에서 직접 웹사이트를 호스팅합니다.
- 장점: 완전 무료, Git과 연동, 간단한 설정
- 단점: 정적 사이트만 가능, 용량 제한(1GB)
- 사용법: 저장소 설정에서 GitHub Pages 활성화

**Netlify**
정적 사이트 전문 호스팅 서비스입니다.
- 장점: 무료 플랜(대역폭 100GB/월), 자동 배포, HTTPS 제공
- 단점: 정적 사이트 위주, 무료 플랜 제한 있음
- 특징: GitHub 연동, 폼 처리, 서버리스 함수

**Vercel**
Next.js를 만든 회사의 호스팅 서비스입니다.
- 장점: 빠른 속도, 자동 최적화, 프리뷰 배포
- 단점: 상업용은 유료 플랜 필수
  - 2025년부터 엄격히 적용
- 특징: 프레임워크 최적화, 엣지 네트워크

## 4. 도메인과 호스팅 연결하기

### 4-1 도메인 구매하기

도메인을 구매하는 과정은 다음과 같습니다:

1. **도메인 등록 업체 선택**
   - 국내: 가비아, 후이즈, 카페24
   - 해외: GoDaddy, Namecheap, Squarespace Domains

2. **도메인 이름 검색**
   - 원하는 도메인이 사용 가능한지 확인
   - 대체 도메인 제안 확인
   - 다양한 TLD 옵션 검토

3. **도메인 등록**
   - 개인정보 입력
   - 등록 기간 선택 (보통 1년 단위)
   - 추가 서비스 선택 (프라이버시 보호 등)

도메인 선택 시 고려사항:
- 짧고 기억하기 쉬운 이름
- 브랜드와 연관성
- 철자가 어렵지 않은 단어
- 하이픈(-) 사용은 가급적 피하기

### 4-2 네임서버 설정

도메인과 호스팅을 연결하려면 네임서버를 설정해야 합니다. 네임서버는 도메인이 어느 서버를 가리킬지 알려주는 역할을 합니다.

**네임서버 변경 과정:**
1. 호스팅 업체에서 네임서버 주소 확인
2. 도메인 관리 페이지 접속
3. 네임서버 설정 메뉴 찾기
4. 호스팅 업체의 네임서버 주소 입력
5. 변경사항 저장 (반영까지 24-48시간 소요)

예시 네임서버 주소:
```
ns1.hosting-company.com
ns2.hosting-company.com
```

### 4-3 DNS 레코드 관리

DNS 레코드는 도메인이 어떻게 작동할지 설정하는 것입니다.

**주요 레코드 타입:**
- **A 레코드**: 도메인을 IPv4 주소와 연결
- **CNAME 레코드**: 도메인을 다른 도메인과 연결
- **MX 레코드**: 이메일 서버 지정
- **TXT 레코드**: 도메인 소유권 확인용

설정은 호스팅 업체의 DNS 관리 페이지에서 레코드 타입과 값을 입력하면 됩니다.


웹사이트를 인터넷에 공개하려면 도메인과 호스팅이 필수입니다. 도메인은 주소, 호스팅은 집이라고 생각하면 됩니다. 최근에는 GitHub Pages, Netlify, Vercel 같은 무료 정적 호스팅 서비스들이 많아져서 개인 프로젝트나 포트폴리오 사이트를 쉽게 배포할 수 있습니다.

도메인 구매는 신중하게 결정하되, 호스팅은 프로젝트 규모와 기술 스택에 맞춰 선택하면 됩니다. 처음에는 무료 호스팅으로 시작하고, 트래픽이 늘어나면 유료 서비스로 이전하는 것도 좋은 전략입니다.
