## 📋 구조 설계 원칙

### SAL 3D Grid 기반 설계
- **X축 (Stage)**: 의존성 중심 - 5~7 단계
- **Y축 (Area)**: 병렬성 중심 - 7~10 영역
- **Z축 (Task)**: 인접성 중심 - 세부 작업

### 핵심 규칙
1. **병렬 가능 작업은 Y축(영역)에 배치**
2. **의존 관계가 있는 작업만 X축(단계)에 배치**
3. 프로세스 ≠ 순차적 12단계
4. Frontend/Backend는 **동시 진행 가능** (같은 Stage, 다른 Area)

## 🎨 프로세스 특징

### ✅ 핵심 원칙
1. **프롬프팅 코딩**: AI에게 프롬프트만, 코드 작성은 0줄
2. **트랙터 운전**: 사용자 = 운전사, AI = 트랙터
3. **프로토타입 = 프로덕션급**: 99% 완성도 목표
4. **실 DB 연결**: 프로토타입부터 Supabase 실제 DB 사용
5. **Grid 우선**: 프로토타입 **전에** Grid 완성

### ⚠️ 주의사항
- **절대 시간 계획 세우지 않음** (사용자 명시)
- Export/Import는 **프로세스가 아닌 작업 방식**
- Frontend/Backend는 **병렬 진행 가능** (순차 아님)
- Stage 수는 **5~7개가 적정** (12개 과도함)

### 🔄 의존 관계
```
기획 → 플로우차트 → 기술스택 선정
                      ↓
            프로젝트 그리드 작성 ⭐
                      ↓
                개발 준비 설정
                      ↓
            프로토타입 제작 (실 DB 연결)
                      ↓
┌─────────────────────┴──────────────────────┐
│                                              │
Frontend 개발        Database/Backend 개발 (병렬)
│                                              │
└─────────────────────┬──────────────────────┘
                      ↓
              테스트 및 검증
                      ↓
              배포 및 운영
```

## 📊 SAL 3D Grid 매핑

### Stage (X축) - 의존성 중심 (5~7개)
1. 기획 및 분석
2. 프로젝트 그리드 작성
3. 프로토타입 제작
4. 코드 구현 (Frontend + Backend 병렬)
5. 테스트 및 검증
6. 배포 및 운영
7. (선택) 고도화 및 피드백

### Area (Y축) - 병렬성 중심 (7~10개)
1. 기획/전략
2. 설계/아키텍처
3. DB/데이터
4. Frontend
5. Backend
6. 디자인/UX
7. QA/테스트
8. 운영/관리
9. 문서/협업
10. AI/자동화

### Task (Z축) - 인접성 중심
- 약 35~50개 Task 카드
- 각 (Stage × Area) 교차점에 배치
- 빈 셀 허용 (모든 셀에 Task 불필요)
