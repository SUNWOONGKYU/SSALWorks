## 개요

### 목적
SSAL Works 플랫폼을 원활하게 운영하기 위해 관리자(써니)가 수행해야 하는 모든 수동 작업을 정리

### 전제 조건
- Admin Dashboard 접근 권한 보유
- Supabase 프로젝트 관리자 권한
- 하나은행 기업 뱅킹 접근 권한
- SendGrid 이메일 관리 권한

### 작업 우선순위
1. 🔴 **긴급 (Urgent):** 즉시 처리 (30분 이내)
2. 🟡 **중요 (Important):** 당일 처리
3. 🟢 **일반 (Normal):** 2-3일 이내 처리

## 일일 업무 (Daily Operations)

### 1. 아침 루틴 (09:00 - 09:30)

#### 1-1. 대시보드 개요 확인

**접속:** Admin Dashboard → 📊 대시보드 개요

**확인 항목:**
- ✅ 총 회원 수 (전일 대비 증감)
- ✅ 미처리 문의 개수 (긴급 건수)
- ✅ 이번 달 매출 (전월 대비 증감)
- ✅ 크레딧 판매량

**예시 화면:**
```
┌─────────────────────────────────────┐
│ 총 회원 수: 1,234명                 │
│ ↑ +15명 (어제 대비)                 │
├─────────────────────────────────────┤
│ 미처리 문의: 12건                   │
│ ⚠️ 긴급 3건                         │
├─────────────────────────────────────┤
│ 이번 달 매출: ₩5,200,000           │
│ ↑ +8% (전월 대비)                   │
└─────────────────────────────────────┘
```

**액션:**
- 긴급 문의 3건 확인 → 즉시 처리
- 전일 대비 회원 급증 시 → 이유 확인 (광고 효과? 이벤트?)

---

#### 1-2. 최근 활동 타임라인 검토

**위치:** 대시보드 개요 → 📋 최근 활동

**확인 항목:**
```
🔴 긴급 문의: 결제 오류 (3분 전)
🟢 신규 가입: 홍길동 (12분 전)
🟡 크레딧 환불 요청 (25분 전)
🟢 구독 갱신 완료 (1시간 전)
```

**액션:**
- 🔴 긴급 항목 → 즉시 처리
- 🟡 중요 항목 → 메모 후 오전 중 처리
- 🟢 일반 항목 → 확인만

---

#### 1-3. 긴급 처리 필요 항목 확인

**위치:** 대시보드 개요 → ⚠️ 긴급 처리 필요

**우선순위 목록:**

| 우선순위 | 항목 | 건수 | 예상 소요 |
|---------|------|------|----------|
| 🔴 High | 미납 회원 (구독 정지 예정) | 3명 | 20분 |
| 🔴 High | 크레딧 환불 요청 | 5건 | 30분 |
| 🟡 Medium | 잔액 부족 회원 | 12명 | 10분 |
| 🟡 Medium | 신규 문의 (미답변) | 8건 | 40분 |

**액션:**
- 각 항목 클릭 → 해당 섹션으로 이동
- 우선순위대로 처리 계획 수립

### 2. 무통장 입금 확인 (09:30 - 10:00)

#### 2-1. 하나은행 기업 뱅킹 확인

**접속:** 하나은행 → 기업 뱅킹 → 입금 내역

**확인 기간:** 전일 09시 ~ 당일 09시

**입금 내역 예시:**
```
2025-12-02 08:35 | 홍길동 | ₩3,000,000 | "SSAL설치비"
2025-12-02 07:12 | 김철수 | ₩50,000 | "크레딧충전"
2025-12-01 18:22 | 이영희 | ₩100,000 | "크레딧충전"
```

---

#### 2-2. Admin Dashboard에서 대기 중인 결제 확인

**접속:** Admin Dashboard → 💳 결제 관리 → 결제 내역

**필터 적용:** 상태 = "대기 중" + 결제 수단 = "무통장입금"

**테이블 예시:**
```
┌──────────────┬─────────────┬────────────┬──────────┬────────┐
│ 결제 ID      │ 사용자      │ 금액       │ 결제수단 │ 상태   │
├──────────────┼─────────────┼────────────┼──────────┼────────┤
│ PAY-2025-045 │ 홍길동      │ ₩3,000,000 │ 무통장   │ 대기   │
│ PAY-2025-046 │ 김철수      │ ₩50,000    │ 무통장   │ 대기   │
│ PAY-2025-047 │ 이영희      │ ₩100,000   │ 무통장   │ 대기   │
└──────────────┴─────────────┴────────────┴──────────┴────────┘
```

---

#### 2-3. 입금 내역 매칭 및 승인

**Step 1: 수동 매칭**

하나은행 입금 내역과 Admin Dashboard 대기 결제를 비교:

| 은행 입금자 | 금액 | Admin 사용자 | 금액 | 매칭 |
|-----------|------|-------------|------|------|
| 홍길동 | ₩3,000,000 | 홍길동 (PAY-2025-045) | ₩3,000,000 | ✅ |
| 김철수 | ₩50,000 | 김철수 (PAY-2025-046) | ₩50,000 | ✅ |
| 이영희 | ₩100,000 | 이영희 (PAY-2025-047) | ₩100,000 | ✅ |

**Step 2: Admin Dashboard에서 승인 처리**

각 결제 건별로:
1. [상세] 버튼 클릭
2. 입금자명, 금액 확인
3. [입금 확인] 버튼 클릭
4. 확인 모달에서 [승인] 클릭

**Step 3: 자동 처리 확인**

시스템이 자동으로 수행:
- ✅ `payment_transactions` 테이블: `status` → `"completed"`
- ✅ 설치비인 경우: `users` 테이블 `installation_paid` → `true`
- ✅ 크레딧 충전인 경우: `credit_balance` 증가
- ✅ 사용자에게 영수증 이메일 자동 발송
- ✅ 플랫폼 내 알림 생성

**Step 4: Toast 알림 확인**

```
✅ 결제가 승인되었습니다.
   - PAY-2025-045 (홍길동, ₩3,000,000)
```

#### 2-4. 매칭 안 되는 입금 처리

**시나리오 1: 은행 입금자명이 다른 경우**

은행: "홍길동아버지" → Admin: "홍길동"

**처리:**
1. 입금 시간, 금액으로 추정
2. 사용자에게 확인 이메일 발송:
   ```
   제목: [SSAL Works] 입금 확인 요청

   안녕하세요, 홍길동님.

   다음 입금 내역이 확인되었으나 입금자명이 일치하지 않습니다:
   - 입금자: 홍길동아버지
   - 금액: ₩3,000,000
   - 입금일시: 2025-12-02 08:35

   본인의 입금이 맞는지 회신 부탁드립니다.
   ```
3. 사용자 확인 후 수동 승인

**시나리오 2: 금액이 다른 경우**

은행: ₩3,010,000 → Admin: ₩3,000,000 (수수료 포함)

**처리:**
1. 금액 차이 확인 (수수료인지, 오입금인지)
2. 수수료인 경우: 승인 처리
3. 오입금인 경우: 사용자에게 확인

**시나리오 3: 대기 중인 결제 없이 입금된 경우**

은행: "박민수" ₩100,000 → Admin: 해당 결제 건 없음

**처리:**
1. 사용자 검색 (이메일, 이름)
2. 사용자 존재 시: 본인 확인 후 크레딧 충전
3. 사용자 없음 시: 환불 처리 (사유: 오입금)

#### 3-3. 답변 작성 프로세스 (긴급 문의 예시)

**문의:** INQ-244 - "결제가 안되는데 확인 부탁드립니다"

**Step 1: [답변하기] 버튼 클릭**

모달 폼 표시:
```
┌────────────────────────────────────────┐
│ 💬 문의 답변                           │
├────────────────────────────────────────┤
│ 문의자: test@test.com                  │
│ 접수일: 2025-12-02 13:15               │
│                                        │
│ [질문 내용]                            │
│ 결제가 안되는데 확인 부탁드립니다.      │
│                                        │
│ [답변 작성]                            │
│ ┌────────────────────────────────────┐ │
│ │                                    │ │
│ │ (에디터)                           │ │
│ │                                    │ │
│ └────────────────────────────────────┘ │
│                                        │
│ [취소]  [저장]                         │
└────────────────────────────────────────┘
```

**Step 2: 문제 원인 파악**

답변 작성 전 확인:
1. 결제 관리 섹션에서 해당 사용자 검색
2. 결제 시도 내역 확인
3. 오류 로그 확인 (Supabase Functions 로그)

**예시 확인 결과:**
- 결제 시도: 2025-12-02 13:10
- 오류 코드: `CARD_DECLINED`
- PG사 응답: "한도 초과"

**Step 3: 답변 작성**

```
안녕하세요, test@test.com님.

문의 주신 결제 오류 건 확인했습니다.

[원인]
결제 시도 시 카드사로부터 "한도 초과" 응답을 받았습니다.
(시도일시: 2025-12-02 13:10)

[해결 방법]
1. 카드사에 연락하여 한도를 확인해주세요.
2. 다른 결제 수단을 이용해주세요:
   - 다른 카드
   - 카카오페이
   - 무통장입금

[무통장입금 안내]
입금 계좌: 하나은행 123-456789-01234 (주식회사 쌀웍스)
입금자명: test (사용자 ID 앞 4자리 포함 필수)

입금 완료 후 자동으로 처리됩니다.

추가 문의사항이 있으시면 언제든지 연락주세요.

감사합니다.
써니 드림
```

**Step 4: [저장] 클릭**

자동 처리:
- ✅ `support_requests` 테이블: `status` → `"answered"`
- ✅ 사용자에게 이메일 자동 발송 (답변 내용 포함)
- ✅ 플랫폼 내 알림 생성
- ✅ Toast 알림: "답변이 저장되었습니다"

다시 문의 주시면 바로 확인하겠습니다.

감사합니다.
```

**확인 항목:**
```
┌──────────┬──────────────────┬──────────┬──────────────┬──────────┐

│ I9J0K1L2 │ lee@test.com     │ 어제     │ 0%           │ 미구독   │
└──────────┴──────────────────┴──────────┴──────────────┴──────────┘
```

#### 4-2. 온보딩 진행 상황 확인

**온보딩 단계:**
1. ✅ 이메일 인증 (필수)
2. ⬜ 프로필 작성 (선택)
3. ⬜ 프로젝트 등록 (필수)
4. ⬜ 구독 시작 (필수)

**문제 사례:**

**사례 1: 온보딩 0% (이메일 미인증)**
- 사용자: lee@test.com
- 가입일: 어제 (24시간 경과)
- 문제: 이메일 인증 안 함

**액션:**
1. 이메일 인증 링크 재발송
2. 인증 독려 이메일 발송:
   ```
   제목: [SSAL Works] 이메일 인증을 완료해주세요

   안녕하세요, SSAL Works입니다.

   가입해주셔서 감사합니다!

SELECT user_id, SUM(incorrect_charge) as refund_amount
FROM credit_transactions

  AND incorrect = true

WHERE users.user_id = affected_users.user_id;

## 부록: 빠른 참조 (Quick Reference)

### 자주 사용하는 SQL 쿼리

**1. 오늘 신규 가입자 조회:**
```sql
SELECT user_id, email, created_at
FROM users
WHERE DATE(created_at) = CURRENT_DATE
ORDER BY created_at DESC;
```

**2. 미답변 문의 조회:**
```sql
SELECT id, user_email, question, created_at
FROM support_requests
WHERE status = 'pending'
ORDER BY created_at ASC;
```

**3. 결제 대기 건 조회:**
```sql
SELECT id, user_email, amount, payment_method
FROM payment_transactions
WHERE status = 'pending'
  AND payment_method = '무통장입금'
ORDER BY created_at ASC;
```

**4. 구독 미납 회원 조회:**
```sql
SELECT u.user_id, u.email, ms.next_payment_date,
       CURRENT_DATE - ms.next_payment_date AS days_overdue
FROM monthly_subscriptions ms
JOIN users u ON ms.user_id = u.user_id
WHERE ms.status = 'unpaid'
ORDER BY days_overdue DESC;
```

**5. 크레딧 잔액 부족 사용자 조회:**
```sql
SELECT user_id, email, credit_balance
FROM users
WHERE credit_balance < 1000
  AND subscription_status = 'active'
ORDER BY credit_balance ASC;
```
