2. Stage 개요

S4 (개발 3차)는 품질 보증 및 최적화 단계입니다.

Stage 목표

  • 결제 시스템 연동
  • 종합 테스트 (Unit, Integration, E2E)
  • 성능 최적화
  • 버그 수정 및 안정화
  • 보안 감사

3. Stage 작업 영역

이 Stage에서는 다음 Area의 작업이 진행됩니다:

| Area | 작업 내용 | 유형 |

|------|----------|------|

| Documentation | 테스트 문서화 | AI-Only |

| Frontend | 결제 관련 UI | AI-Only |

| Backend_Infra | 결제 시스템 모듈 | Human-AI |

| Backend_APIs | 결제 API, 웹훅 처리 | Human-AI |

| Security | 결제 보안 검증 | AI-Only |

| Testing | Unit/Integration 테스트 | AI-Only |

| DevOps | 성능 최적화 | AI-Only |

상세 Task 목록은 Order Sheet를 참조하세요.

4. 작업 규칙

파일 저장 규칙 (이중 저장)

  • Frontend, Backend_APIs → Stage 폴더 + Production 폴더
  • Testing, Documentation → Stage 폴더에만

새 폴더 생성 금지

기존 폴더 구조 내에서만 작업

5. 실행 프로세스 (10단계)

```

[Step 0] Order Sheet 저장

[Step 1] Project SAL Grid 확인 - Stage Task 목록 조회

[Step 2] Task 실행 순서 결정 - Dependencies 기반

[Step 3] 각 Task 실행 - Task Agent 서브에이전트 투입

[Step 3-1] Human-AI Task 처리 (해당 시)

[Step 4] 결과물 저장 - Stage/Area 폴더 + Production 이중 저장

[Step 5] 검증 - Verification Agent 서브에이전트로 검증

[Step 6] Grid 업데이트 - Status, Files, Progress 기록

[Step 7] Stage Gate - 모든 Task 완료 후 Stage Gate 검증

[Step 8] PO 기능 테스트 요청 - 최종 승인

```

핵심 원칙:

  • Task Agent ≠ Verification Agent (작성자와 검증자 분리)
  • Grid가 진실의 원천: 모든 Task 정보는 Project SAL Grid에 있음
  • 서브에이전트 투입: Task Agent, Verification Agent 사용

6. PO(Project Owner) 필수 작업

Backend_Infra 영역

  • PG사 가맹점 등록 (심사 기간 소요)
  • API 키 발급 (테스트용/프로덕션용)

  • 웹훅 URL 설정

Testing 영역

  • 테스트 실행 및 결과 확인
